CS21120 Worksheet 5: Creating class diagrams for the CVM

 

Introduction

This exercise will help you understand the notation used in class diagrams, the process used during the Analysis workflow to construct them from a use case description and the use of Rose to draw class diagrams.

This will be achieved by realising the Obtain Confectionery Item use case of the CVM.

Steps

  1. Use the technique of noun identification (and complementary questions) to select a candidate set of classes from the Obtain Confectionery Item use case description, and any related inclusion or extension use case descriptions.
  2. Find any further candidates by asking specific questions. See below.
  3. Whittle down this list of candidates by asking refinement questions, listed below. Explicitly record rationale for excluding candidates since this will form a useful audit that you may want to come back to later on.
  4. Find any further classes by following the Unified Process recommendations: identifying entity, boundary and control classes, see below.
  5. Add these classes to your Rose logical view package with a brief description of the role that each plays in the realisation. Do not add attributes or operations at present.
  6. Use the technique of verb and verb-phrase identification to list candidate associations and their kind (e.g. normal, containment, aggregation etc). This tends to be less useful than the noun-identification technique. Most of the time you will have to use your judgement when finding associations. If objects of two classes need to send each other messages, then you will probably need an association between them.
  7. Whittle down this list of candidates by asking refinement questions, as listed below. Again, explicitly record rationale for excluding candidates.
  8. Draw the analysis-level class diagram for the Obtain Confectionery Item use case using selected classes and associations. Name the class diagram something like: Obtain Confectionery Item Realisation-Analysis, to make it clear what the purpose of the class diagram is. Note that Rose does not support the analysis class stereotype icons. Instead you will have to use normal class elements with appropriate stereotype labels. You may find that you need to add further classes or associations that were not identified earlier or exclude some. This is a creative process and there are usually many ways to realise a use case, each with its own merits but not necessarily better than others. Inevitably, you will have to make several attempts before you produce a diagram that you are happy with.
  9. Identify candidate attributes by looking for nouns, adjectives and possessive phrases. Some attributes may have already been identified during class rationalisation.
  10. Whittle down this list by asking refinement questions (below). Include only those attributes that are required for the Obtain Confectionery Item use case realisation.
  11. Assign attributes to the classes they naturally belong to, using the high cohesion rule.
  12. If you can identify opportunities for factoring out common attributes then you might like to define super-classes and use generalisation associations.

Techniques for finding and refining classes, associations and attributes

Use the following questions to help you identify potential classes, associations and attributes and also to whittle down the candidates based on certain criteria. Do this when examining a use case description that you wish to realise as a class diagram.

You will also want to apply the guidelines suggested by the Unified Process to help identify analysis classes (also attached below). Try and combine all the techniques when constructing your analysis class diagram realisation for the Obtain Confectionery Item use case.

 

Noun identification technique for finding classes

Look for nouns in the use case description & in the general problem domain.

In addition, you might find a few more classes by asking the following specific questions:

  • Do we have information to be stored, transformed or handled?
  • Are there any external actors that need representing internally? You might need to maintain some local state about an actor, in which case you need a class.
  • Do we have any patterns, class libraries and components from earlier projects to reuse? For this exercise you won’t find this question very useful. In a real project you might find this useful.

Refining the candidate list of classes

The following questions will help you remove inappropriate classes.

  • Are there any redundant classes? That is, do you have several classes, perhaps with different names, that are really the same class?
  • Are there any classes out of scope? For example, Account and Customer in the ATM System example. Classes that really belong to other systems and not the system you are modelling.
  • Are there any vague classes, and are you using the most descriptive role names? Make your class names as specific as possible. Make sure every class has a useful role in the use case class diagram realisation.
  • Are there any attributes masquerading as classes? Simple values should be represented as attributes and not classes. E.g. The car is a hatchback. "Hatchback" is a noun but would be an attribute of the class car (called type) and not a class in its own right.
  • Are there any operations masquerading as classes? This can conflict with Unified Process control classes, since these tend to have operation names. Apart from these special control classes, other classes with names that appear to be operations should probably be changed to be analysis responsibilities (assigned as a responsibility statement in an appropriate class).
  • Is the class really a design or implementation entity? You’re trying to create an analysis model, not a design model. Make sure you’re only representing problem space entities.
  • Is the class really a meta-language concept? For the CVM an example would be "Simulation". This would not be a class because it’s a property of the CVM system.

Verb identification technique for finding associations

Examine the use case description for verb phrases. These may turn into associations, e.g.

  • those indicating physical juxtaposition of objects (e.g. entity1 next to entity2),
  • directed actions (e.g. entity1 drives entity2),
  • communication events (e.g. entity1 sends message to entity2),
  • ownership (e.g. entity1 contained in entity2), etc.

Here the phrases in italics would map to candidate associations between the classes entity1 and entity2. If you’ve got containment you might want to use an aggregation or composition association. You might also want to use a qualified association if that adds useful extra information to the class diagram.

Questions to help rationalise the candidates:

  • Are there any associations between eliminated classes? If so then you don’t need these associations.
  • Are there any misnamed associations? You use the most descriptive names. You don’t have to use the names you found in the use case description.
  • During design workflow avoid associations, the links of which would be short-lived. That is, think how you would implement the class in code. If the association is to be implemented as a class member variable then it is long-lived. If the association between two classes is temporary, e.g. an object reference sent as a parameter and used temporarily by a method, then don’t represent this as a UML association between these classes.

Noun/adjective/possessive phrase technique for finding attributes

  • Candidate attributes and the classes they belong to can be found by looking for nouns, adjectives and possessive phrases in the use case description.

Questions can help us rationalise the candidates:

  • Is the attribute really a class? If it can have attributes of its own then it is probably a class.
  • Is it really a qualifier attribute rather than a class attribute? For example, you might decide to move the ISBN attribute from a Book class to the qualified association between a Browser class and the Book class. You should try and avoid duplicating the attribute in several places.
  • Is the attribute really an association? During analysis, make sure that references between classes are represented by associations and not attributes.
  • Is the object class attribute really an association class attribute? If you find that an attribute is duplicated across several classes and there is a many to many association between them, then it might be better to define an association class that contains the attribute and is attached to the many to many association.
  • Does the attribute belong to the class (high cohesion)? Only logically related attributes should be combined.

Unified Process Guidelines for finding analysis classes

  • For entity classes, study use case description and any domain model in detail to identify passive information.
  • Identify one main boundary class for each human actor.
  • Identify one primitive boundary class for each entity class found earlier.
  • Identify one main boundary class for each external system actor.
  • Identify one control class that handles coordination of use case realisation.
